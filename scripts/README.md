# 映画館データ収集パイプライン

「消えた映画館の記憶」(https://hekikaicinema.memo.wiki/, CCライセンス・サイト名表示条件) を
一次情報源として、営業中の映画館を抽出し、OpenStreetMap(Overpass)で閉館・座標・公式サイトを
クロスチェックした上で `api/data/curated/` に都道府県（東京は区）単位のJSONとして追加する。

このドキュメントだけ見れば、新しい地域の追加・既存データの更新に迷わず着手できることを
目標にしている。全国展開作業（2026年）自体の経緯・詳細ログは
[`progress-archive.md`](./progress-archive.md)、現在の到達点サマリーは
[`progress.md`](./progress.md) を参照。

進行順（全国展開時の実績）: 関東 → 中部 → 関西 → その他 → **完了**（2026年7月）

---

## 1. パイプライン全体像

```
[wiki (hekikaicinema.memo.wiki)]
        │  1. scrape-wiki.js（EUC-JP取得・営業中/閉館リスト抽出）
        ▼
  {operating: [...], closed: [...]}
        │  2. 実在検証（WebSearchで手動、営業中リストの信頼性を担保）
        ▼
   検証済み営業中リスト
        │  3. geocode.js（Nominatim: POI名→番地→丁目→町名フォールバック）
        ▼
   座標付きJSON（precision: poi / neighbourhood / failed）
        │  4. add-website-links.js（Wikipedia infoboxから公式サイト取得）
        ▼
   website付きJSON（未取得分はWebSearchで手動補完）
        │  5. check-osm-closed.js（Overpassで実座標を取得し突合）
        │     - 閉館未反映の検出（wikiの閉館リスト × OSM現存タグ）
        │     - websiteのOSMタグ優先差し替え候補の抽出
        │     - 座標乖離チェック（curated座標 vs OSM実座標）
        ▼
   最終検証済みJSON
        │  6. 保存・動作確認・品質ゲート
        ▼
  api/data/curated/<地域>.json（コミット）
```

`api/data/closed-cinemas.json` は上記フローとは別に、「OSM側のタグが未更新のまま残っている
閉館館」だけを都度追記していく単一ファイル（`api/cinemas.js`が検索結果から除外するために使う）。

## 2. スクリプト一覧

| スクリプト | 役割 | 入力 | 出力 |
|---|---|---|---|
| `scrape-wiki.js` | wikiページ（EUC-JP）を取得し、営業中／閉館の館名・住所を抽出。「開館前」表記も検出して警告 | wikiページ名（複数可） | `{operating: [], closed: []}` のJSON |
| `geocode.js` | 住所→座標のジオコーディング（Nominatim） | scrape-wiki.jsの出力 | 座標・precision付きJSON |
| `add-website-links.js` | Wikipedia infoboxから公式サイトURLを取得 | geocode.jsの出力 | website付きJSON |
| `check-osm-closed.js` | Overpassで対象地域の実タグを取得し、閉館突合・website差し替え候補・座標乖離を出力 | scrape-wiki.js出力＋geocode.js出力 | 目視確認用のレポート（標準出力） |

すべて対象地域単位（都道府県または市区町村ページ群）で実行する。1回のバッチで複数の
wikiページをまとめて処理してよい。

## 3. 運用上の注意（ハマりどころ）

- **devサーバーの再起動が必要**: `api/cinemas.js` は `api/data/curated/` の全JSONをサーバー起動時に
  一度だけ読み込む。ファイル追加後にローカル確認する場合は、devサーバーを再起動しないと反映されない。
- **closed-cinemas.jsonへの追加は必ず目視確認**: `check-osm-closed.js` の`[closed]`候補はnameMatch()の
  部分一致による誤検出があり得る（例: 「イオンシネマ座間」が閉館館「米ケ浜シネマ座」の一部「シネマ座」に
  誤って一致、「京成ローザ10」（営業中）が「京成ローザ」（1999年閉館）に誤って一致）。機械的に採用せず、
  営業中リストとの重複がないか・WebSearchで実際の閉館日等を毎回確認する。
- **Windowsのシェル文字コード**: このマシンのシェルはcp932（Shift-JIS）で、日本語文字列を
  inline引数（`node -e "..."`や生の`curl`）として渡すと文字化けすることがある。日本語文字列を
  扱うロジックは必ず`.mjs`スクリプトファイルとして書き、実行する。

## 4. 実行手順（1地域＝1バッチのランブック）

作業ファイルは一時ディレクトリ（`$SCRATCH` など）に置き、リポジトリには最終JSONだけを入れる。

### 0. 対象ページの洗い出し

`scripts/wiki-page-list.txt` に、wikiの全ページ一覧（`https://hekikaicinema.memo.wiki/l/`）から
抽出した「〜の映画館」ページ323件のマスターリストがある。対象都道府県の市区町村名でgrepして
対象ページを確定させる（トップページ・ランディングページを都度たどるより確実）。

```
grep -E "新潟市|長岡市|..." scripts/wiki-page-list.txt
```

### 1. wikiから抽出

```
node scripts/scrape-wiki.js 新宿区の映画館 $SCRATCH/scraped.json
```

- ページ名はwikiトップページの**リンクテキスト**と完全一致させる（ページタイトルと異なる場合が
  ある。例:「北海道地方の映画館」ではなく「北海道の映画館」、「大館市の映画館」ではなく「大館市」）。
  大きいバッチを流す前に対象ページ名を`wiki-page-list.txt`と突き合わせて事前検証すること
  （スクレイプはページ全件成功して初めて出力する実装のため、途中の1件でも名前を誤ると
  それまでの取得分ごと失われる）。
- 複数ページを1バッチにまとめる場合はページ名を並べる
- `[要確認] 所在地が取れていません` が出た館は元ページを目視確認
- 「開館前」表記の警告が出た館は要注意（後述「既知の落とし穴」参照）

### 2. 実在検証（重要）

wikiの「営業中」は編集が古い可能性がある。営業中リストの各館について:

- 公式サイトが取れている館 → URLが生きているか確認
- 取れていない館 → WebSearchで「<館名> 公式サイト」を検索し、営業状況と公式サイトを確認
- どこにも情報がない館 → 閉館・移転を疑い、確認できるまで curated に入れない
  （渋谷区の「HACK」のような正体不明データを混入させないため）

注意: 複数館が同居していたブロック（例: 渋谷TOEI→Bunkamuraル・シネマ）は、
wikipediaTitle が旧館のものになっていることがある。website付与前に目視確認する。

### 3. ジオコーディング

```
node scripts/geocode.js --region 東京都新宿区 $SCRATCH/scraped.json $SCRATCH/geocoded.json
```

- POI名 → 番地住所 → 丁目 → 町名 の順にフォールバック。全失敗は `precision: "failed"`
- `failed` の館は Googleマップ等で座標を手動確認して補う（precisionは実態に合わせ poi / neighbourhood）
- `neighbourhood` はアプリ上で「おおよその位置」表示になる
- **ブランド名のみのPOI名検索は要注意**（後述「既知の落とし穴」参照）。特に「イオンシネマ」
  「109シネマズ」等、支店名を含まない名前で検索した場合は、結果が対象地域内の座標か必ず確認する

### 4. 公式サイト付与

```
node scripts/add-website-links.js $SCRATCH/geocoded.json $SCRATCH/with-website.json
```

- wikiに公式サイトがあった館（`website` が既にある）はそのまま
- Wikipedia infoboxから取れなかった館は WebSearch で補完し手動で追記
- Wikipedia infoboxの外部リンクが入居施設（モール等）を指していることがある（例:
  ヒューマントラストシネマ有楽町でitocia.jp＝モールのサイトが誤って入った事例）。
  次のOSM突合ステップでOSM側のwebsiteタグがあれば優先して差し替える。

### 5. OSM閉館突合・website優先順位の適用・座標検証

```
node scripts/check-osm-closed.js --region 東京都新宿区 $SCRATCH/scraped.json $SCRATCH/geocoded.json
```

出力の見方（自動で確定させず、必ず目視確認する）:

| ラベル | 意味 | やること |
|---|---|---|
| `[ok]` | 営業中と名前一致 | なし |
| `[closed]` | wikiの閉館館と名前一致 | WebSearchで裏取りしてから `api/data/closed-cinemas.json` に追加 |
| `[要確認]` | 閉館・営業中の両方に一致（旧名タグの現役館 or 同名の別館） | 座標・閉館日で判断 |
| `[ok?]` | 近接のみ一致（名前不一致） | 英字表記の同一館か、近くの別の閉館館かを確認 |
| `[unknown]` | どれとも一致しない | WebSearchで実在確認。確認できなければ closed-cinemas.json で除外 |

#### website解決ルール（優先順位）

**OSMのwebsiteタグ（実座標に紐づく） > wiki本文/Wikipedia infoboxの公式サイト（モール等の
誤リンクの可能性あり） > null。**

`[ok]`（名前一致）した館でOSM側にwebsiteタグがあれば、末尾に「websiteをOSM側の値に差し替える
候補」として出力されるので、目視確認のうえ適用する。差し替え候補はURLのスラッグと地名を
照合して機械的に検証できる（例: `cinema.aeoncinema.com/wm/kuwana/` の「kuwana」が「桑名」と
一致するか）。

モールドメインらしきURL（施設全体のサイトで、個別の上映スケジュールに辿り着けないもの。
イオンモール・アリオ・パルコ・ららぽーと等）しか無くOSMにもタグが無い場合は、無理に埋めず
**nullのままにする**（誤ったリンクを出すより「無し」の方が安全）。null化した館は
`scripts/website-todo.json` に `{name, address, file, previousWebsite, reason}` の形で記録し、
後日の個別調査に回す（自動処理はしない）。

#### 閉館判定ルール

`[closed]`候補は**機械的に採用しない**。`nameMatch()`は部分一致(includes)のため、ブランド名や
略称が異なる館同士で誤って一致することがある（例:「イオンシネマ座間」が「米ケ浜シネマ座」に、
「京成ローザ10」（営業中）が「京成ローザ」（1999年閉館）に誤って一致）。候補ごとに
WebSearchで実際の閉館日・跡地情報を確認してから`closed-cinemas.json`に追加する。

#### 座標検証ルール

名前一致した館について、curated側の座標とOSM実座標の距離が2000mを超えると「座標がOSM実座標と
2000m以上離れている館」として警告される。ジオコーディング誤り（住所の町名レベルフォールバックが
誤った地点にヒットする等）の早期発見に使う。ブランド名のみで支店名を持たないOSMタグ（「イオン
シネマ」「109シネマズ」等）は同一県内の複数館すべてと部分一致してしまうため、`nearestByName`
（名前一致した候補のうち最も近い1件を採用）で誤ペアリングを防いでいるが、それでも偽陽性が出る
ことがある。

**重要**: website差し替え候補と違い、座標の差し替え候補には機械的な検証手段がない。
`nearestByName`は「名前一致した候補の中で一番近いもの」を選ぶだけなので、ブランド名のみの
OSMタグが閉館済み店舗と現役店舗の両方に部分一致する場合、たまたま近い方＝正しい方とは限らない
（実例: 三重県のイオンシネマ津南で、「イオンシネマ」という汎用タグが閉館済み「イオンシネマ津」の
座標を指しており、機械的に採用すると誤った場所に移動してしまった）。**座標の差し替え候補は
住所とWebSearchで必ず裏を取ってから適用する。機械的な採用は禁止。**

同時に、curated側同士の同一sourcePage内で15km以上離れた組がないかもチェック対象（広域の
都道府県ページに複数都市が混在しているだけの正常なケースと、実際のジオコーディング誤りを
区別するため、こちらも目視確認が必要）。

### 6. 保存と動作確認

1. 検証済みデータを `api/data/curated/<地域>.json` として保存（例: `tokyo-shinjuku.json`）
   - 必須フィールド: `name`, `address`, `lat`, `lon`, `precision`, `website`
   - `wikipediaTitle`, `sourcePage` は残してよい（APIはクライアントに送らない）
2. ローカルdevで対象地域の主要駅を検索し、件数・名前・徒歩分・公式サイトリンクを確認
   ```
   npm run dev
   curl "http://localhost:5173/api/cinemas?q=新宿駅"
   ```
   （devサーバーは起動時に一度だけcuratedデータを読み込むため、追加後は再起動すること）
3. `npm run build` が通ることを確認

### 7. 品質ゲート（コミット前チェック）

- [ ] 件数: 映画館情報サイト（eiga.com等の地域別リスト）の掲載数と大きく乖離していない
- [ ] 全座標が対象地域内にある（geocode.jsのbboxチェックで担保、failedは手動解決済み）
- [ ] 主要駅検索で重複・閉館館が出ない
- [ ] precision の `poi` 比率を確認（`neighbourhood` ばかりなら住所抽出の質を疑う）
- [ ] website null化・差し替えの根拠が`website-todo.json`／コミットメッセージに残っている

### 8. コミット

1バッチ＝1コミット。ユーザー確認後にpush（Vercel自動デプロイ）。

## 5. 既知の落とし穴

- **「開館前」表記の罠**: wikiが「開館前」（＝閉館リスト扱い）のまま実際は開業済みの
  ケースがある（例: TOHOシネマズ名古屋栄、イオンシネマ八王子滝山）。`scrape-wiki.js`の
  検出ロジックで必ずチェックし、該当があればWebSearchで開業状況を確認する。
- **ブランド名のみタグのあいまい一致**: 「イオンシネマ」「109シネマズ」「TOHOシネマズ」等、
  支店名を含まないOSMタグは全国の複数の実在地に存在する。`check-osm-closed.js`は
  `nearestByName`で対策済みだが、**`geocode.js`のPOI名検索には同等の対策が未実装**
  （既知の未修正課題。イオンシネマ広島・イオンシネマ綾川で実際に誤座標を発見・手動修正した
  実績あり）。ブランド名のみで検索した結果は必ず対象地域内かどうか確認する。
- **政令指定都市の行政区再編**: 浜松市など行政区名が変更された都市では、Nominatimの
  町名フォールバックが無関係な地点にヒットすることがある（実例: 浜松市「中区」表記で
  11〜33km離れた誤地点がヒット）。区再編があった都市（浜松・堺・熊本・新潟等）は
  同一sourcePage内の距離チェックを特に注意して見る。
- **wikiの市区町村カバレッジは不完全**: 座間市・平塚市・羽生市など、wiki自体がページを
  持たない市区町村が全国に多数ある。これらの館はOverpassマージのみで検索結果には出るが、
  curated化（website・precisionの付与）はされていない。新規に対象を広げる際は
  `wiki-page-list.txt`との突合で「本当にページが無いのか」を確認できる。
- **同一住所で異なる館名の重複**: 同一建物・複合施設内の複数スクリーン（例: 丸の内ピカデリー
  ／DOLBY CINEMA、TOHOシネマズ梅田本館／別館）は正当な別エントリ。座標重複チェックで
  引っかかっても即誤りと判断せず、実在確認すること。
- **一般公開されていない施設**: 在日米軍基地内シアター、プラネタリウム・公民館ホールへの
  `amenity=cinema`誤タグは追加対象外。

## 6. 補助ファイルの使い方

| ファイル | 用途 | 更新方法 |
|---|---|---|
| `scripts/wiki-page-list.txt` | wiki全ページ一覧（323件）のマスターリスト。対象地域のページ名確定に使う | 手動更新は基本不要（wiki側の構成変更時のみ再取得） |
| `scripts/website-todo.json` | website未確定（null）の館を`{name, address, file, previousWebsite, reason}`で記録した個別調査待ちリスト | バッチ処理中にnull化した館を追記。個別調査はClaudeが自動で行わず、後日ユーザーが対応 |
| `api/data/closed-cinemas.json` | OSM側のタグが未更新のまま残っている閉館館（`api/cinemas.js`が検索結果から除外するために参照） | `check-osm-closed.js`の`[closed]`候補をWebSearchで裏取りしてから追記。`matchNames`は小文字化した名前バリエーションの配列、`lat`/`lon`はその閉館館自体の座標（除外判定は座標150m以内＋名前完全一致でゲートされる） |
| `api/data/curated/<地域>.json` | 最終成果物。都道府県（東京は3分割）単位 | 本ランブックの手順で追加・更新 |
| `scripts/progress.md` | 全国展開の現在地サマリー | ブロック完了時や大きな仕様変更時に更新 |
| `scripts/progress-archive.md` | 全国展開作業の詳細な生ログ（時系列） | 新規の大規模バッチ作業を行う場合は同様の形式で追記可 |

## 7. レート制限（スクリプトに組み込み済み）

| サービス | 制限 | 実装 |
|---|---|---|
| wiki (seesaawiki) | 良識の範囲 | ページ間2秒待機・UAに連絡先明記 |
| Nominatim | 1リクエスト/秒 | 1.1秒待機 |
| Wikipedia | 良識の範囲 | 500ms待機 |
| Overpass | 混雑時504 | 地域あたり1クエリ・504時は15秒後に1回リトライ |

いずれも無料の共有インフラ。バッチを連続実行する場合も1バッチずつ、検証を挟みながら進める。
